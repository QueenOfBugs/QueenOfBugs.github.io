使用Github遇到的问题记录

github actions

git会记录文件状态

使用 Emacs 编辑文件的情况下: 很容易一边编辑文件,另一边就打开了shell-mode 要 对刚刚的更改进行提交. 当一个文件在本地打开,没有保存时,他会处于被其他程序占用的状态,如果这时候进行 了 git 提交的话,该文件的状态一样会被记录,当项目使用了 github action 并且进行 的操作涉及到这个未保存的文件,那 action 就会失败,提示该文件被某个用户或程序锁 定.

生成和添加 SSH keys 到github

生成新的 SHH 密钥

ssh-keygen -t ed25519 -C "wj915614874@163.com"

添加 SSH 密钥到 ssh-agent

后台启动 ssh 代理

eval "$(ssh-agent -s)"

将 ssh 私钥添加到代理

ssh-add ~/.ssh/id_ed25519

将 SSH 密钥添加到 Github 帐号

将下面命令的结果复制到 https://github.com/settings/ssh/new 里的 Key 表单中. 

cat ~/.ssh/id_ed25519.pub

测试 SSH 连接

ssh -T git@github.com

如果前面都设置正常的,应该会提示你验证公钥是否正确,输入 yes 后会看到类似下面的信 息: 

Hi username! You've successfully authenticated, but GitHub does not provide shell access.

通过代理连接 SSH

因为经常被墙的原因,使用代理连接 SSH 是很有必要的.这里去分一下 git 命令里设置的走 代理, git 里设置的代理是用来 clone 下载文件的,走的是 http/https 协议,因为下载文 件的网址也时不时被墙,或者直连下载极慢,所以需要走代理;而这里使用的 SSH 协议是用来 验证我们自己的账户,保证账户安全的,还可以方便我们对自己的仓库进行操作, 这两个是 不同的,所以不要以为设置了 git 的代理就可以使用代理进行 ssh 连接了. openssh 自身是不带代理功能的(网上说的,具体是不是没有翻文档),需要使用另外的工具来 实现代理,这里需要的是 nc(netcat) ,这是一个强大的网络工具,但 gnu-netcat 这个版本 的并没有提供代理功能,我的 Manjaro-i3 已经安装了 nc, 不过是 gnu-netcat 版的,实验 多次还以为哪里有问题,直到看到这篇博文 通过代理连接 SSH 里提到 openbsd-netcat 版 本的netcat才有代理功能,才爬出这个坑.具体步骤就不再详细写了,那篇博文写的很清楚了, 记录下使用的命令和配置:

  • 使用命令完成代理: 

    ssh user@server -o "ProxyCommand nc -X 5 -x 127.0.0.1:1080 %h %p"

  • 使用 ".ssh/config" 配置文件: 

    Host *
  ProxyCommand=nc -x 127.0.0.1:1080 %h %p

Github.io 踩坑

路径含有隐藏文件夹时url访问失败

如果网页资源的某个文件路径中有隐藏文件(夹)，那该资源就会访问失败.

Windows 上会的仓库会自动修改可执行文件的权限,且 chmod 没效果

我部署blog 的脚本里有一个 build.sh 文件,在 linux 上正常可以执行,等我在 windows 上克隆了仓库,并做出修改,push 到远程仓库,就发现 github action 失败,原因是 build.sh 没有权限,察看权限后发现 build.sh 的可执行权限没了. windows 上用 chmod +x build.sh 但没有效果,换回 linux 环境,发现文件权限竟然改了,改回去之后 提交到仓库, github action 正常,再切换回 windows 环境,pull 下来代码,啥都还没干, git status 察看状态,就发现 build.sh 权限给改了,莫名其妙,暂时不知道具体原因,也 不想排查根本原因,找到了一个 有效 的办法:

  • 查看文件权限: 

    git ls-files --stage build.sh

  • 修改权限: 

    git update-index --chmod=+x build.sh

  • 再次查看权限: 

    git ls-files --stage build.sh

事实证明,上面的方法在显示时确实可以显示权限修改了,但实际上在 window 上 push 上传 到仓库后还是失败,报措仍然是权限问题.

原因是什么呢?是因为 windows 上的 git 其实并不保存文件的实际权限,它只是 "模拟" 了 文件的权限,一般的文件只有读写权限的,不会出问题,而像 shell 脚本这种有可执行权限的 文件它的模拟权限就搞不定了,特别是 shell 脚本还是 linux 系统上的可执行文件,它只识 别出读和写的权限,git 就检测出和仓库中文件权限的不一致了,所以在 windows 上提交后 就会导致修改文件权限. 要避免这样的问题出现, 可以让 git 忽略文件权限的变化,这样 windows 上的 git 就只会追踪和提交文件内容的改变,而不会影响权限了. 命令行配置: 

git config core.filemode false

直接修改配置文件: 

# file : .git/config
# This is the config file, and
# a '#' or ';' character indicates
# a comment
#

; core variables
[core]
	; Don't trust file modes
	filemode = false

关于 filemode 的详细解释见文档: core.filemode

远程仓库强制覆盖本地仓库.

适用于本地未提交的修改,但远端仓库已经从另一台机器提交了新的修改.如果本地未提交的 修改并不重要,可以删除,那么就使用这种方法: 

git fetch --all
git reset --hard origin/main
git pull

fetch 从远程下载最新代码,但不执行合并或rebase 等操作. reset 将主分支重置为 fetch 下载的内容. hard 选项更改工作树中所有文件以匹配 origin/main 中的文件.
